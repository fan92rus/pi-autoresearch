# PRD — parallel-режимы для pi-autoresearch

**Статус:** Draft v0.1
**Дата:** 2026-07-23
**Автор:** fan92rus
**Upstream:** [davebcn87/pi-autoresearch](https://github.com/davebcn87/pi-autoresearch) v1.6.1, база — форк [fan92rus/pi-autoresearch](https://github.com/fan92rus/pi-autoresearch) (Windows spawn fixes)

---

## 1. Кратко

**Parallel-режимы** расширяют pi-autoresearch (этот форк) тремя режимами распараллеливания проверки гипотез через субагентов: один агент идёт последовательно (как сегодня), но этап «сформулировать гипотезу → применить правку → измерить → решить keep/revert» исполняется **N параллельных веток одновременно** в изолированных git worktree. Родитель-оркестратор агрегирует результаты, переизмеряет победителя (борьба с selection bias) и ведёт канонический лог.

Дополнительно вводится **жёсткий бюджет времени эксперимента** (5 минут по умолчанию): если `measure.sh` превышает бюджет, цикл не запускается вообще — вместо этого агент получает задачу «сделай эксперимент быстрым или уменьши покрытие, чтобы он вписывался в бюджет и при этом оставался репрезентативным».

---

## 2. Проблема

Текущий autoresearch последователен: одна правка → один замер → keep/revert → следующая. Это означает:

1. **Длинный time-to-answer на разреженном пространстве.** Большинство гипотез — мусор; последовательный перебор тратит стеночные часы на тупики.
2. **Нет эксплуатации ортогональности.** Правка парсера и правка рендера независимы — их можно было бы проверить и **стекать** одновременно, а не одну за другой.
3. **Нет поиска, только жадная оптимизация.** Текущая стратегия — локальный жадный спуск. При многоэкстремальном ландшафте (ML-метрики, сборщики, perf) она застревает в локальном минимуме.
4. **Стоимость растёт линейно, выигрыш — тоже линейно.** Дешёвые flash-модели и избыточные ядра простаивают, пока один агент ждёт завершения тяжёлого `measure.sh`.

## 3. Цели (Goals)

| # | Цель | Метрика успеха |
|---|------|----------------|
| G1 | **Режим A — Best-of-N:** параллельная проверка N гипотез для одной цели, выбор лучшей | Корректно выбирает победителя ≥ 90% случаев на синтетическом бенчмарке (где истинный оптимум известен) |
| G2 | **Режим B — Orthogonal stacking:** декомпозиция на независимые подсистемы, параллельная проверка, стекающий merge | Стек из K ортогональных правок даёт суммарное улучшение ≈ Σ индивидуальных (отклонение < 15%) |
| G3 | **Режим C — Space search (beam/bandit):** поддержка K живых состояний, ветвление, отсев | На синтетическом мультимодальном ландшафте находит глобальный оптимум чаще, чем жадный режим (на синтетике с известным ответом) |
| G4 | **Бюджет времени эксперимента:** время одного `measure.sh` ≤ 5 мин (настраивается) | Нулевая ситуация «эксперимент крутится 20 минут»; при превышении — agent steer «сократи/перепиши» |
| G5 | **Сохранение контракта:** существующие `.auto/`-сессии и sequential-режим продолжают работать | Обратная совместимость: старый `/autoresearch on` поведение идентично upstream |
| G6 | **Изоляция:** параллельные ветки не ломают ни канонический лог, ни git main | Тест гонки (N параллельных писателей) не повреждает `log.jsonl` |

## 4. Не-цели (Non-Goals)

- **Распределённый кластер по сети.** Параллелизм — локальный (одна машина, git worktree, процессы pi-субагентов). Не Kubernetes.
- **Автоматическая декомпозиция произвольной задачи.** Режим B требует, чтобы цель была декомпозируема на независимые подсистемы; оркестратор помогает, но не гарантирует декомпозицию там, где её нет.
- **Замена sequential-режима.** Параллельные режимы — опциональны. Простые задачи остаются на sequential.
- **Реальный ML-training search.** Хотя режим C архитектурно поддерживает beam search, это не замена Ray Tune / Optuna для гиперпараметрического поиска обучения моделей.
- **GUI/веб-дашборд для нескольких сессий.** Существующий дашборд расширяется минимально; многоклиентский обозреватель — вне скоупа.

## 5. Пользовательские сценарии (User Stories)

### US-1: Best-of-N для одной узкой цели
> Как разработчик, оптимизирующий скорость парсера, я хочу, чтобы autoresearch **одновременно** попробовал 4 разных подхода к одной и той же горячей функции (инлайнинг, таблица переходов, SIMD, мемоизация) и выбрал лучший — чтобы не ждать 4 последовательных цикла.

**Принятие:** `/autoresearch parallel-best-of-n 4 optimize parser hot path` → 4 worktree, 4 worker, parent выбирает победителя, cherry-pick в main, канонический лог содержит все 4 + итоговый.

### US-2: Стек ортогональных правок
> Как разработчик, уменьшающий время сборки, я хочу параллельно проверить «кеш зависимостей», «параллельный линкер», «tree-shaking» — потому что они ортогональны и я хочу все три.

**Принятие:** `/autoresearch parallel-stack [deps-cache, parallel-linker, tree-shaking]` → 3 worktree от общего baseline, каждый измеряет свою правку, parent применяет их последовательно к main с re-measure после каждой, финальный лог показывает накопленный эффект.

### US-3: Space search для мультимодального ландшафта
> Как ML-инженер, тюнящий loss, я хочу, чтобы autoresearch поддерживал «несколько живых гипотез» одновременно и отсевал тупики, вместо того чтобы углублять один путь.

**Принятие:** `/autoresearch parallel-search --beam-width 3` → поддерживает 3 лучших состояния, на каждом шаге генерирует M кандидатов, оценивает параллельно, оставляет топ-3. Лог показывает дерево探索.

### US-4: Защита от долгого эксперимента
> Как любой пользователь, я не хочу, чтобы autoresearch крутил 20-минутный бенчмарк в каждом из 4 параллельных воркеров, сжигая 80 минут CPU.

**Принятие:** При `measure.sh` > бюджета (5 мин default) run прерывается, статус = `budget_exceeded`, агент получает steer: «эксперимент превысил бюджет; увеличь производительность measure.sh или перепиши его так, чтобы он покрывал целевую задачу и был быстрым». Цикл продолжается с исправленным measure.sh.

## 6. Требования

### 6.1 Функциональные

| ID | Требование | Приоритет |
|----|------------|-----------|
| FR-1 | Режим A: запуск N worker-субагентов в N worktree от общего baseline commit | P0 |
| FR-2 | Каждый worker: применяет гипотезу, вызывает `run_experiment`, возвращает `{diff, metric, metrics, notes, commit?}` | P0 |
| FR-3 | Parent: ранжирует результаты по median-метрике, отсеивает шум (Δ < noise floor) | P0 |
| FR-4 | Parent: **переизмеряет** победителя на main перед keep (selection bias correction) | P0 |
| FR-5 | Parent: cherry-pick / apply diff победителя, `log_experiment(keep)` в канонический лог | P0 |
| FR-6 | Режим B: parent/planner декомпозирует цель на независимые подсистемы | P1 |
| FR-7 | Режим B: параллельная проверка каждой подсистемы в worktree | P1 |
| FR-8 | Режим B: parent применяет топ-K ортогональных правок последовательно, re-measure после каждой | P1 |
| FR-9 | Режим C: beam search — K живых состояний, M кандидатов на шаг, отсев до топ-K | P2 |
| FR-10 | Режим C: опционально UCB/ε-greedy exploration policy | P3 |
| FR-11 | Бюджет: `run_experiment` имеет `budget_seconds` (default 300) | P0 |
| FR-12 | Бюджет: при превышении — статус `budget_exceeded`, steer к агенту | P0 |
| FR-13 | Бюджет: steer содержит конкретные варианты (reduced fixture, fewer iterations, `BENCH_MODE=quick`) | P1 |
| FR-14 | Канонический лог: parent — единственный писатель `.auto/log.jsonl` в main worktree | P0 |
| FR-15 | Worker-логи: каждый worktree имеет свой `.auto/log.jsonl` (изолирован через workingDir) | P0 |
| FR-16 | Очередь гипотез: `.auto/queue.jsonl` — parent пишет, workers читают | P1 |
| FR-17 | Консолидация: parent merge'ит worker-логи в канонический после раунда | P1 |
| FR-18 | Backward compat: sequential `/autoresearch on` поведение идентично upstream | P0 |

### 6.2 Нефункциональные

| ID | Требование |
|----|------------|
| NFR-1 | **Изоляция состояний:** гонка при N параллельных писателей не повреждает канонический лог (тест `concurrency/log-race`) |
| NFR-2 | **Идемпотентность rerun:** повторный запуск того же раунда детерминирован (при фиксированном measure.sh) |
| NFR-3 | **Лимит ресурсов:** `concurrency` по умолчанию = min(CPU-1, 4); настраивается |
| NFR-4 | **Нулевая блокировка main:** parent не держит git lock main дольше одного cherry-pick |
| NFR-5 | **Прозрачность:** пользователь видит в дашборде все N параллельных веток и их статусы |
| NFR-6 | **Windows-совместимость:** Git Bash для всех shell-операций (унаследовано от базы-форка) |

## 7. Бюджет времени эксперимента (отдельно, т.к. в требовании пользователя)

**Проблема:** параллелизм умножает стоимость. Один 20-минутный `measure.sh` × 4 worker = 80 минут CPU (или стеночные, если нет ядер). Это убивает смысл распараллеливания.

**Контракт:**
- `run_experiment` получает параметр `budget_seconds` (default **300** = 5 мин).
- При превышении: процесс kill, результат = `status: "budget_exceeded"`, `metric: null`.
- Агент получает steer:
  ```
  ⏰ Эксперимент превысил бюджет 300с (фактически 312с).
  Варианты:
  1. Уменьшить покрытие: fixture/iter count так, чтобы замер был репрезентативен и < 300с
  2. Включить BENCH_MODE=quick в measure.sh (подмножество нагрузки)
  3. Закэшировать дорогие setup-шаги между запусками
  Перепиши measure.sh и повтори. НЕ продолжай с долгим экспериментом.
  ```
- `measure.sh` контракт: поддержка env-переменной `BENCH_MODE={full|quick|smoke}` для градации нагрузки.
- Режимы parallel (A/B/C) наследуют бюджет и **не запускают worker**, если baseline-замер уже превышает бюджет (сначала почини).

## 8. Метрики успеха продукта

- **Time-to-improvement:** время от старта сессии до первого keep с confidence ≥ 2.0×, сравнение sequential vs parallel-A (целевое ускорение 1.5–3× на разреженных задачах).
- **Coverage:** количество проверенных гипотез за фиксированный бюджет стеночного времени (целевое ≥ 2× для parallel-A vs sequential).
- **Stuck-rate:** доля сессий, застрявших в локальном оптимуме без улучшений > 5 итераций (целевое снижение для parallel-C vs sequential на мультимодальных задачах).
- **Budget compliance:** 100% запусков вписываются в бюджет (или корректно триггерят steer).
- **Log integrity:** ноль повреждённых `log.jsonl` при параллельных раундах (тест + прод).

## 9. Риски и митигации

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Selection bias при best-of-N (выбор лучшего из шумных оценок) | Высокая | Среднее | Переизмерение победителя на main + confidence score; продвигать только при Δ > noise floor |
| Параллельные git операции ломают общий `.git` | Средняя | Высокое | Worktree имеют отдельные index; parent — единственный коммитер в main; cherry-pick сериализован |
| `measure.sh` недетерминирован → шум на параллельных замерах | Высокое | Высокое | Median-of-3 в каждом worker; mandatory `BENCH_MODE` контракт; переизмерение победителя |
| Стоимость токенов растёт с N | Средняя | Среднее | concurrency лимит; flash-модели для workers; только измерение в worker, оркестрация в parent |
| Worktree creation накладные расходы на больших репо | Низкая | Низкое | Переиспользование worktree между раундами; shallow clone опционально |
| Конкуренция за CPU снижает реальный параллелизм | Высокая | Среднее | concurrency = min(CPU-1, 4); advisory в дашборде |

## 10. Скоуп v0.1 (MVP)

MVP включает:
- ✅ Режим A (Best-of-N) — P0 требования
- ✅ Бюджет времени (FR-11, FR-12)
- ✅ Parent-only committer, изоляция worktree
- ✅ Backward compat

MVP **не** включает (отложено):
- ⏳ Режим B (Orthogonal stacking) — после A
- ⏳ Режим C (Space search) — после B
- ⏳ Очередь гипотез `.auto/queue.jsonl` — после A
- ⏳ UCB/ε-greedy — после C

**Обоснование последовательности:** A — самый простой и даёт немедленную ценность; B требует декомпозиции (дороже); C — исследовательский, нужен только при подтверждённой узости A. Бюджет — общий фундамент для всех.
